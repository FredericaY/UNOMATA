# UNOMATA 游戏设计文档

> 本文档为副线接龙玩法的权威设计参考，所有实现以此为准。

---

## 一、核心设计理念

主线（TPS竞技场）与副线（骇入接龙）同时运行，互不暂停。
玩家需要主动分配注意力——在射击/躲避的同时决定何时发起骇入、骇入哪个目标、是否值得冒险完成完整接龙。

骇入的本质是**降低目标敌人的伤害减免**，而非强化玩家自身武器。效果只作用于被骇入的那个敌人：
- 未被骇入的敌人：拥有极高伤害减免（约95%），几乎无法打出有效伤害
- 骇入中/骇入结算有效期内的敌人：伤害减免按接龙结果比例降低，满档时无减免甚至附加负减免（受到伤害加成）
- 骇入满后继续接：溢出部分转化为生命回复技能充能

这一机制带来的核心决策：**先骇入谁？** 面对多个敌人时，玩家需要判断集中骇入最危险的目标，还是分散骇入快速清场。

---

## 二、主线：TPS竞技场

### 基础规则
- 无限波次竞技场，每波消灭所有敌人后进入下一波
- 核心操作：移动、瞄准、射击、躲避
- 敌人数量少、机动性低，场景简单

### 难度曲线
| 波次 | 敌人血量 | 攻击频率 |
|------|---------|---------|
| 早期 | 低 | 低 |
| 中期 | 中 | 中 |
| 后期 | 高 | 高 |

### 伤害减免机制
敌人自带基础伤害减免，骇入是降低该减免的唯一手段（见副线联动章节）。

## 三、副线：骇入接龙

### 3.1 主题包装
接龙机制以"骇入"为科幻主题包装，视觉表现为全息能量面板，不直接呈现 UNO 牌面。
牌面重新设计为科幻风格的能量符文/频率信号，保留颜色、数字、特殊功能语义。

### 3.2 触发流程（参考《PRAGMATA》骇入机制）

```
1. 玩家瞄准有效距离内的敌人
2. 按下骇入键（默认 F / 手柄 LB）
3. 系统检测准星是否命中有效目标
4. 命中 → 发送骇入波（特效）→ 显示骇入UI，接龙开始计时
5. 未命中 → 无反应（不消耗冷却）
6. 接龙结束 → 判定奖励 → UI关闭
```

- 骇入目标：准星命中范围内最近的单个敌人
- 骇入期间：玩家仍可移动和射击，但无法切换骇入目标
- **同时只能有一个骇入进行中（v1限制）**，已有骇入进行时按键无效
- 多目标同时骇入为后续扩展方向，当前版本预留接口但不实现
- 骇入期间敌人不被"锁定"，仍可正常攻击玩家

### 3.3 卡池构成

简化版 UNO 卡池，共 **48 张**（不含王牌，王牌走独立概率通道）：

| 类型 | 数量 | 明细 | 备注 |
|------|------|------|------|
| 数字牌 | 40 | 4 色 × 数字 0–9 | 有色，有数字 |
| 反转牌 | 8  | 4 色 × 2 张      | 有色，**无数字** |
| 王牌   | —  | 不进 deck         | **无色，无数字**；由 `WildAppearRate` 独立概率塞入 |

颜色集合：`{ Red, Blue, Green, Yellow }`

> 卡池总量是发牌算法 `pick_random_legal` / `pick_random_illegal` 的"理想牌堆"。
> 王牌**不进 deck**，每轮独立按 `WildAppearRate` 决定是否塞入选项；
> Number 与 Reverse 按 `SolvableRate` 决定是否抽 1 张合法牌、其余位填非法牌。
> 具体抽样算法见 `INTERFACE.md` 第五节"发牌算法"。

### 3.4 接龙基础规则

每次会话有一个**总倒计时**。倒计时内玩家连续接牌：

- 显示**当前牌**（CurrentCard）+ 若干**选项牌**
- 玩家从选项中选出能接上的牌
- 选中合法牌 → 已接数 +1，所选牌成为新的 CurrentCard，下一轮选项刷新
- 选中非法牌 → 处理见 3.7 "结算与扣血规则"
- 倒计时归零 → 接龙结束，按当前已接数量结算（不扣血）

### 3.5 匹配规则（核心改动 — 严格 ±1 升序/降序）

会话维护一个**接龙状态**：

```
state.lastColor   ∈ { Red, Blue, Green, Yellow, null }
state.lastNumber  ∈ { 0..9, null }
state.direction   ∈ { Ascending, Descending }   // 初始 Ascending
```

会话**开局**：`lastColor = null, lastNumber = null, direction = Ascending`，
等价于"桌面上有一张空牌（Empty）作为 CurrentCard"。开局这张 null 牌**承接规则等同王牌后状态**——任意牌都可作为首张接出。

#### 3.5.1 候选牌（next）合法性判定

| next 类型 | 合法条件 |
|-----------|---------|
| **王牌**     | 永远合法（任何状态下都能接） |
| **反转牌**（颜色 C'）| 同色（C' == lastColor）**或** lastColor == null |
| **数字牌**（颜色 C', 数字 N'）| 满足以下任一即可：<br>① `lastColor == null`：开局或刚接王牌后，**任意数字合法**<br>② 同色：`C' == lastColor`（数字不限）<br>③ 异色 + 严格 ±1：`lastColor != null` **且** `lastNumber != null` **且** （Ascending 时 `N' == lastNumber + 1`，Descending 时 `N' == lastNumber - 1`） |

> **关键规则变更**：旧版"同色 OR 同数字"已废除，且旧版"严格大于/小于"也已收紧。
> 数字接龙改为**严格连续 +1 升序或严格连续 -1 降序**，异色合法解唯一。
> 异色同数字（Blue-5 接 Red-5）非法；异色非连续（Blue-7 接 Red-5 升序）也非法。
>
> **特别注意**：刚接反转牌后（`lastColor != null` 且 `lastNumber == null`）→
> **异色数字全部非法**，只能接同色 Number / 同色 Reverse / Wild。
> 这与"开局 / 王牌后"的 `lastColor == null` 状态形成对照：
> 开局/王牌后 lastColor 与 lastNumber 同时为 null，任意合法；
> 反转后 lastColor 仍保留，但 lastNumber 已清，方向约束因无基准而**异色路径关闭**。

#### 3.5.2 当前牌（prev）对状态的更新

玩家选中合法牌后，立即用该牌更新 state：

| prev 类型 | 更新方式 |
|-----------|---------|
| **数字牌**（颜色 C，数字 N） | `lastColor = C; lastNumber = N; direction 不变` |
| **反转牌**（颜色 C） | `lastColor = C; lastNumber = null; direction 翻转` |
| **王牌** | `lastColor = null; lastNumber = null; direction 不变` |

> 反转牌**只切换方向并保留颜色基准**，不携带数字基准；其后第一张同色数字牌的数字成为新基准。
> 王牌**完全重置**颜色和数字基准，**不**改变方向（对应"王牌不改变基于数字的接龙顺序"）。
>
> **反转牌连接策略**：每色提供 2 张 Reverse，玩家可在合适时机连续打出两张同色 Reverse
> 来连续翻转方向（第一张后 `lastColor` 仍为该色，第二张同色 Reverse 仍合法）。

#### 3.5.3 边界与示例

```
开局: state=(null, null, Asc)  CurrentCard=Empty （等价王牌后）
玩家接 Red-5  → ✓ lastColor=null，任意合法
            state=(Red, 5, Asc)
玩家接 Red-2  → ✓ 同色（异色升序需 N'==6，但同色覆盖）
            state=(Red, 2, Asc)
玩家试探 Blue-5 → ✗ 异色 + 严格升序需 N'==3，5≠3
玩家接 Blue-3 → ✓ 异色 + 严格升序 N'==2+1
            state=(Blue, 3, Asc)
玩家接 Blue-Rev → ✓ 同色反转
            state=(Blue, null, Desc)
玩家试探 Yellow-2 → ✗ 异色 + 反转后 lastColor=Blue ≠ null + lastNumber=null
                  → 异色路径关闭，全部非法
玩家接 Blue-7 → ✓ 同色（同色覆盖方向）
            state=(Blue, 7, Desc)
玩家接 Wild → ✓ 王牌永远合法
            state=(null, null, Desc)  // 方向不变，lastColor/lastNumber 全清
玩家接 Red-9 → ✓ 王牌后 lastColor=null，任意合法
            state=(Red, 9, Desc)
玩家接 Red-Rev → ✓ 同色反转（每色 2 张，第一张消耗）
            state=(Red, null, Asc)   // 方向再翻
玩家接 Red-Rev → ✓ 同色反转（第二张消耗，连续两张同色 Reverse 合法）
            state=(Red, null, Desc)
玩家接 Red-4 → ✓ 同色任意
            state=(Red, 4, Desc)
```

#### 3.5.4 选项呈现顺序（必须随机化）

每轮生成的 `OptionCount` 张选项牌，**呈现给玩家前必须经过随机洗牌**，使合法牌、非法牌、王牌的位置在 UI 上完全不可预测。

**理由**：
- 若位置固定（如"合法牌总在第 0 位"），玩家可绕过认牌直接按位置点击，`SolvableRate` 概率系统失去意义，副线退化为反应测试。
- 死局判定（`isDeadlock`）也依赖玩家**先看牌再决策**的认知路径，位置泄漏会让死局奖励机制（3.10.3）失效。

**实现约束**：
- 洗牌发生在合法/非法/王牌三类牌全部填入选项数组之后、返回给 Unity 端之前
- 洗牌使用同一个注入的 `Random` 源（便于测试固定 seed 重放）
- 洗牌后 `isDeadlock` 标志只看选项内容（"是否存在任一合法牌"），与位置无关

> 详见 `INTERFACE.md` 第五节"发牌算法"伪代码 `shuffle(options)` 步骤。

### 3.6 难度参数

| 参数 | 早期 | 后期 |
|------|------|------|
| 选项数量（OptionCount） | 3 选 1 | 5 选 1 |
| 总倒计时（TotalTime） | 长 | 短 |
| 轮级有解率（SolvableRate） | 高 | 低 |
| 满档基础所需接龙数（TargetChainCount / basePot） | 少 | 多 |

> "拿满奖励所需接龙数"是核心压力来源：时间更短 + 需要接更多张，双重压缩玩家的操作余量。
>
> `SolvableRate` **不**直接由波次决定，而是由玩家的**同步率（SyncRate）**决定（见第七章）。
> 后期波次仍可在 `OptionCount` / `TotalTime` / `TargetChainCount` 三个维度提升压力。

### 3.7 奖励池：basePot / maxPot 双层模型

**核心概念**：奖励上限不是固定的 `TargetChainCount`，而是动态扩展的 `maxPot`：

```
basePot = config.TargetChainCount        // 固定基线，对应 100% 伤害去除
maxPot  = basePot
        + 反转牌使用次数 × 1
        + 王牌使用次数   × 4
```

每接到一张反转牌 / 王牌时，`maxPot` 即时增长（满档前）。
**反转牌让 maxPot +1，王牌让 maxPot +4**——这是鼓励玩家"消耗特殊牌冲击更高伤害"的设计。

#### 3.7.1 满档触发（OnMaxReached）

```
单次合法接牌的处理顺序：
  1. chain += 1
  2. 牌效结算（更新 state，未满档时 maxPot 增长）
  3. 满档判定: 若 chain >= maxPot 且未满档过 → 进入满档状态
       - isMaxLatched = true（单向 latch，不可回退）
       - maxPot 冻结（之后再用反转/王牌不再增长）
       - 触发 OnMaxReached 事件
  4. 溢出判定: 若已满档且 chain > maxPot_frozen → overflow += 1，触发 OnOverflow
```

**满档是单向状态**：进入后，即便后续操作上看 `chain` 似乎"少于"某个动态值，
也不会回到未满档。进入满档时的 `maxPot` 即为本次会话的最终奖励上限。

#### 3.7.2 惩罚机制（待平衡）

> **本节为占位说明**。具体惩罚数值与触发条件需在 Phase 5 测试后敲定，Phase 1 仅按"扣固定血量"占位实现。

会话结束的几种触发：

| 触发 | 是否扣血 | 是否影响 SyncRate | 备注 |
|------|---------|-----------------|------|
| **接错牌**（`OnChainFailed`） | **待平衡** | **待平衡** | 候选方案：仅扣血 / 仅掉同步率 / 两者叠加 |
| **时间到**（`OnTimeUp`） | 不扣血 | 不影响 | 自然结束，按当前 chain 结算 |
| **主动弃牌**（`Surrender`） | 不扣血 | 见 3.10 死局奖励 | 任何状态下玩家按弃牌键都合法 |
| **已满档后任何中断** | 不扣血 | 不影响 | 满档后无负担 |

候选惩罚版本（Phase 5 三选一）：

```
版本 A (温和):
  接错不扣血, 仅 SyncRate 大幅下降
  扣血责任完全交给 TPS 主线 (受伤掉同步率, 同步率影响副线难度)
  → 双线职责清晰

版本 B (中庸):
  接错扣固定血量 + SyncRate 下降
  保留双线惩罚

版本 C (严苛):
  接错仅扣血, SyncRate 不变
  → 早期版本写法, 不推荐
```

设计意图：
- 没接到 basePot 之前，错误是真实代价
- 一旦满档，玩家"无负担继续冲"——错也罢、超时也罢，奖励一定到手
- "推迟满档"（接特殊牌将 maxPot 抬高）= 主动追求更高 factor，**纯奖励**

#### 3.7.3 伤害减免削减系数

```
DamageReductionFactor = ChainCount / basePot     // 不 clamp，无上限
```

示例（basePot = 10）：

| 实际接龙 | 含特殊牌 | factor | 含义 |
|---------|---------|--------|------|
| 0  | -                   | 0.0 | 减免 0 削减 → 敌人保持 95% 减免 |
| 5  | -                   | 0.5 | 减免削减 50% |
| 10 | 全数字              | 1.0 | 满档 → 敌人减免归零 |
| 12 | 含 2 反转           | 1.2 | 满档 + 20% 受击加成 |
| 16 | 含 1 王牌 + 2 反转  | 1.6 | 满档 + 60% 受击加成 |

- **效果只作用于被骇入的目标敌人**，其他敌人不受影响
- 骇入加成效果持续一段固定时间后**清零**（不衰减，直接归零）
- 下次骇入结算时用新系数覆盖，每次骇入独立计算，不累积

### 3.8 溢出奖励：生命回复技能充能

**只有满档之后**继续接到的合法牌才算溢出（即 `chain > maxPot_frozen` 之后每张 +1 充能）：

```
每次满档后接到合法牌 → overflow += 1 → 玩家生命回复技能充能 +1
```

- 溢出充能不再影响 factor（满档时 maxPot 已冻结）
- 任何牌（含反转、王牌）超过 maxPot_frozen 后都算 1 次充能
- 生命回复技能可缓存若干次充能（具体上限待平衡调整）
- 玩家主动触发回复时消耗充能，恢复一定血量
- 设计意图：满档之后纯奖励，鼓励熟练玩家"接到底"

### 3.9 Combo 机制（预留，暂不实现）

以下为后续扩展方向，当前版本只保留接口（`ComboType` 枚举），不实现逻辑：
- 连续两次同色 → 时间延长
- 连续两次符合方向数字 → 额外伤害加成
- 更多 Combo 类型 TBD

### 3.10 死局机制 + 弃牌

#### 3.10.1 死局判定

每轮发牌后，Core 判定本轮是否**死局**：

```
isDeadlock = 选项中无任一合法牌（且不含王牌，因为王牌永远合法）
```

死局是发牌算法的自然产物——`SolvableRate < 1.0` 时必然偶发死局，详见 `INTERFACE.md` 第五节"发牌算法"。

> **死局发生范围**：仅在中盘状态 `(C, N, *)`（即 `lastColor != null` 且 `lastNumber != null`）
> 与反转牌后状态 `(C, null, *)` 下可能触发。
> 开局 / 王牌后 `(null, null, *)` 状态因 `lastColor == null` 任意牌合法，
> 非法池为空、合法位扩展守卫触发，**isDeadlock 恒为 false**。

#### 3.10.2 弃牌按键

骇入键（默认 F）**复用为弃牌键**：

- 无骇入中按 F → 触发骇入
- 骇入中按 F → 立即弃牌（`HackSession.Surrender()`）
- 启动后 0.2~0.3 秒内的 F 输入忽略，防误触

弃牌不扣血，按当前 chain 结算 `DamageReductionFactor`，触发 `OnSessionEnd(EndReason.Surrender)`。

#### 3.10.3 死局快速判断奖励

设计意图：奖励玩家的认知速度。看到死局 → 立即弃牌 = 高手；磨蹭 = 系统帮退。

```
Unity 端死局反应窗口逻辑（数值由 Unity 端常量决定）：
  收到 OnNewRound(..., isDeadlock=true):
    deadlockStartTime = 当前时间
    UI 红色边框警告
  
  玩家在 DEADLOCK_WINDOW_SEC 内主动按弃牌键:
    SyncRate += DEADLOCK_BREAKTHROUGH_REWARD
    session.Surrender()
  
  超过 DEADLOCK_WINDOW_SEC 玩家未操作:
    Unity 主动 session.Surrender()
    无奖励
```

机制特性：
- **永远有兜底**：玩家什么都不做，系统 2 秒后帮退，不会卡死
- **奖励高手**：能在窗口内做出判断的玩家额外赚 SyncRate
- **逻辑自洽**：高手开局靠死局突破积累 SyncRate，无需依赖地图道具
- **滚雪球**：低 SyncRate 玩家死局多 → 突破机会也多 → 反向救赎

#### 3.10.4 解耦说明

`DEADLOCK_WINDOW_SEC` 与 `DEADLOCK_BREAKTHROUGH_REWARD` 都是 **Unity 端**常量，Core 完全不感知。
Core 只负责：
1. 通过 `OnNewRound` 第三个参数 `isDeadlock` 告知 Unity 本轮是否死局
2. 提供 `Surrender()` API（玩家或 Unity 计时器都可调用）

## 四、同步率系统（SyncRate）

> 战术资源，独立于副线接龙规则；与 TPS 主线深度耦合，副线"消费"它。

### 4.1 定义

`SyncRate ∈ [0, 1]`，表示玩家与系统的**同步程度**（叙事包装可改）。

| 范围 | 命名占位 | 含义 |
|------|---------|------|
| 0    | "失同步" | 副线最难，死局频发 |
| 0.5  | "半同步" | 中位，体感"略偏难" |
| 1.0  | "全同步" | 副线几乎稳过 |

初始值：`0.5`（中位起点，符合"成长曲线"叙事；具体值 Phase 5 平衡）。

### 4.2 影响副线难度

```
SolvableRate = 0.5 + 0.45 × SyncRate     // 线性映射

  SyncRate = 0    → SolvableRate = 0.50  → 死局率约 47.5%
  SyncRate = 0.5  → SolvableRate = 0.725 → 死局率约 26%
  SyncRate = 1.0  → SolvableRate = 0.95  → 死局率约 4.75%
```

`SyncRate` 唯一作用即**决定每次骇入会话的 `SolvableRate`**：

```
触发骇入时:
  config.SolvableRate = 0.5 + 0.45 × syncRateModel.Value
  config.WildAppearRate = 0.05  (暂定常量)
  config.OptionCount / TargetChainCount / TotalTime  按波次决定
```

> Core 完全不感知 `SyncRate`，只接收 `SolvableRate` 数值。耦合点 = "new HackSession 那一行的映射公式"。

### 4.3 SyncRate 变化来源

| 来源 | 影响方向 | 数值（Phase 5 平衡） |
|------|---------|---------------------|
| **拾取道具**（地图掉落） | + | 单次固定增量，待平衡 |
| **击杀敌人**（按概率掉落道具） | + | 数值 + 掉落率待平衡 |
| **玩家受到伤害** | − | **方式 c**：按伤害量 / 玩家最大血量比例下降 |
| **死局快速突破**（见 3.10.3） | + | 单次固定增量，待平衡 |
| **副线接错牌** | − ? | 待平衡（参见 3.7.2 惩罚机制候选） |

> 受伤的方式 c 选取理由：重击掉很多、擦伤掉一点，与体感一致。

### 4.4 滚雪球设计

正向螺旋（高手）：
```
不受伤 → SyncRate 高 → 副线易 → 拿到伤害加成
       → 杀敌快 → 拾道具 → SyncRate 更高 → 越滚越大
```

负向螺旋（低手）：
```
受伤 → SyncRate 低 → 副线变难 → 死局多
                                ↓
                           死局突破 ← 兜底救济（见 3.10.3）
                                ↓
                           SyncRate 回升机会
```

设计意图：
- 鼓励"不受伤"作为玩法核心目标，而非简单数值堆砌
- 给低手保留**死局突破**这一翻盘窗口，避免雪球失控滑向地狱
- 同步率与扣血**双重惩罚**让"受伤"在两条线都有反馈

### 4.5 实现解耦

| 层 | 职责 |
|----|------|
| **Core** | 不可见 SyncRate；只接 `config.SolvableRate` |
| **Unity Gameplay** | 持有 `SyncRateModel`，处理拾取/受伤/击杀对它的影响 |
| **Unity Linking** | 触发骇入时把 `SyncRate` 映射成 `SolvableRate` 传入 config；监听 `OnNewRound` 的 `isDeadlock` 启动死局窗口；按弃牌时机决定是否给 SyncRate 奖励 |
| **Unity UI/HUD** | 显示当前 SyncRate（叙事包装的能量条/同步指示器） |

## 六、双线联动

骇入效果**绑定在目标敌人身上**，不是全局状态。每个敌人各自维护自己的骇入状态。

| 敌人骇入状态 | 对该敌人的影响 |
|------------|--------------|
| 未被骇入 / 骇入效果已过期 | 保持基础减免（约95%），几乎无法有效输出 |
| 骇入进行中 | 维持上次骇入系数，骇入结算后更新 |
| 骇入结算：`factor < 1.0`（未满档） | 减免按 factor 比例削减，持续固定时间后清零 |
| 骇入结算：`factor == 1.0`（刚满档无特殊牌） | 减免完全清除，持续固定时间后清零 |
| 骇入结算：`factor > 1.0`（满档+特殊牌加成） | 减免清除，并附加 `(factor - 1.0)` 比例的额外受击加成 |
| 接错牌触发（未满档） | 按 3.7.2 惩罚机制处理（待平衡）；骇入中断结算，目标敌人按当前 chain 获得 factor |
| 接错牌触发（已满档） | 不扣血、不掉同步率；骇入中断结算，目标敌人按 chain_at_break 获得 factor |
| 主动弃牌（含死局突破） | 不扣血；若在死局窗口内则 SyncRate +奖励；目标敌人按当前 chain 获得 factor |
| 满档后继续接龙 | 玩家生命回复技能充能 +1（不影响目标敌人 factor，maxPot 已冻结） |

**核心战术决策点**：
- 先骇入哪个敌人？（优先威胁最大的还是最容易击杀的？）
- 当前骇入值不值得完成？（敌人快死了，骇入收益变低）
- 满档后是继续冲充能，还是断接专心躲避？
- 看到反转/王牌出现，是否值得"冒险走数字下一张"换取更高 maxPot？
- 看到死局了，是抢奖励（快速弃牌赚 SyncRate），还是磨蹭让系统帮退？
- TPS 压力过大时是否中断骇入专心躲避？是否值得为了维持 SyncRate 主动避免受伤？
